# StudentAge Editor Plus — 编辑器修复说明

> BepInEx 补丁插件，修复 StudentAge（学生时代）自带 MOD 编辑器的若干缺陷。
> 取代已废弃的 `StudentAgeModEditorFix`。

- **插件名 / GUID**：`StudentAge Editor Plus` / `com.studentage.editorplus`
- **源码**：`_modsrc/StudentAgeEditorPlus/`
- **产物**：构建后自动部署到 `BepInEx/plugins/StudentAgeEditorPlus/StudentAgeEditorPlus.dll`
- **构建**：`dotnet build -c Release`（游戏需先关闭，否则 DLL 被占用拷不进去）
- **环境**：.NET SDK 8.0，netstandard2.0，Harmony

## 1. 场景事件触发机制失灵（✅ 已修复）

### 现象
MOD 作者反馈：部分事件的触发机制失灵，**"场景"类事件必须手动编辑 JSON 才能触发**，在编辑器里配不出来。

### 根因（已在代码 + 运行时双重验证）
- 场景"进入/离开"事件的触发**编码在事件的 `type` 字段里**，而不是 `mapId`：
  `CommonEvtMgr.ShowSiteEvent()` 计算 `进入 = 900 + 地图号`、`离开 = 800 + 地图号`，再查 `evtDict[该type]`（事件按 `type` 分组）。
  基础游戏的场景事件确实是 `type=90x/80x` 且 `mapId=null`。
- 编辑器里 `type` 是**受限下拉框**（`CfgPropertyRange=EvtTypeCfg`，`ModNormalEditView.OnRenderProperty` 在有 range 时隐藏文本框、**不允许手填**），下拉项只来自 `Cfg.EvtTypeCfgMap`。
- 而 `EvtTypeCfgMap` 只为地图 **{1-8, 11, 17}** 定义了场景类型。其余可触发场景事件（`MapCfg.type==0`）的地图缺类型 → 下拉框选不到 → 只能改 JSON。

### 运行时验证（真实配置，含 DLC，65 条 EvtTypeCfg）
诊断（`SceneTriggerDiagnostic`，默认关闭，配置项 `Diagnostics.RunSceneTriggerDiagnostic` 开启）确认：16 张 `type==0` 地图中，有 **6 张**缺进入/离开类型：

| 地图 | 名称 | 缺失 |
|------|------|------|
| 9 | 鹅城博物馆 | 进入 909 / 离开 809 |
| 10 | 黄金海滩 | 进入 910 / 离开 810 |
| 12 | 鹅城客运站 | 进入 912 / 离开 812 |
| 101 | 新开发区 | 进入 1001（离开撞号，见下） |
| 102 | 南城名苑 | 进入 1002 |
| 103 | 老城区 | 进入 1003 |

### 修复（`SceneEventTypePatch`）
Harmony **Prefix** 挂在 `ModNormalEditView.InitFields`，在编辑器构建 `type` 下拉前，对每个 `type==0` 的地图**幂等补全**缺失的场景类型到 `EvtTypeCfgMap`：
- 进入 `900+地图号`：一律补；
- 离开 `800+地图号`：仅在地图号 < 100 时补（否则 `800+10x ≥ 900` 会与低编号地图的"进入"撞号——这是原版编码本身的限制）。

原生下拉框读的就是这张表，于是这些选项**自动出现**，无需自绘 UI（也复用了游戏原生字体，中文正常）。

实测日志：
```
[SceneEventType] 已为缺失场景补全 9 个事件类型: 809,810,812,909,910,912,1001,1002,1003
```

### 重要：对玩家零依赖
补全的类型只存在于**作者当前编辑会话**。产出的 mod 只是把事件 `type` 存成 `9xx`，在**原版客户端可正常触发**（`ShowSiteEvent → evtDict[9xx] → ShowEvent`，全程不查 `EvtTypeCfgMap`，已逐行验证）。玩家无需安装本插件。

### 边界
- 地图 101/102/103 **只能配"进入"事件**（type 1001-1003）；"离开"因原版编码撞号无法表达。

## 2. mapId 字段语义提示（✅ 已加）

`mapId`（场景）字段**不驱动**"进入场景触发"，只是某些随机事件的筛选器，但它在编辑器里长得像"把事件绑定到场景"，容易让作者误用。

`MapIdHintPatch`（Postfix on `InitFields`，仅 EvtCfg）给该字段的悬浮说明追加一句提示：
> 此字段仅用于筛选，不会让事件在进入该场景时触发。要让事件在进入/离开某场景时触发，请改用上方『类型』字段选择「场景名(进入)」或「场景名(离开)」。

## 3. CG 编辑器：分组 / 编号不可见（✅ 已修复）

CGCfg 走通用表编辑器 `ModNormalEditView`（分发逻辑 `ModPageUploadView`：Person→人物编辑器、Ending→结局编辑器、其余→通用编辑器）。该编辑器只渲染带 `[CfgProperty]` 且非 `Hide` 的字段，于是：
- **`group`（组别）**：原本带 `Hide=true` → 被隐藏；
- **`idx`（编号）**：原本无 `[CfgProperty]` → 根本不在编辑器。

### 这两个字段的作用（`CGLibraryView`）
- **group = CG 在收藏画廊的分类**（`group<0` 隐藏）；另外 `group==0||1` 决定 CG 能否被当背景选用。
- **idx = 画廊里的排序编号**（按 idx 排序；`idx>0` 时显示成 001/002… 三位数）。

所以作者之前根本无法组织/编号自己的 CG。

### 修复（`CgEditorFieldsPatch`）
Postfix on `InitFields`（仅 CGCfg）：
- 解除 `group` 的隐藏（`attr.Hide=false`），并补充取值说明；
- 注入 `idx` 字段项（合成一个 `[CfgProperty]`）；
- 调 `itemgroup_property.SetDatas` 刷新。

实测：CG 编辑界面已出现可填的「组别」和「编号」字段。

## 4. 编辑器：不显示图片预览（✅ 已修复）

### 现象
通用编辑器（`ModNormalEditView`）右侧面板 `group_right` 原本**完全不显示图片**（属性单元格 `Cell_ModNormalPropertyItemUI` 结构里就没有 Image 组件，`group_right` 只有悬浮说明文本）。

受影响的配置类型（均走通用编辑器，原版无预览）：
- CGCfg（CG 图）、BgCfg（背景图）、ItemCfg（物品图标）、BookCfg（书籍封面）、
- RenshengguanMemoryCfg（珍贵记忆配图）、ActionCfg（行动图标）、
- PersonStateCfg（状态图标）、KZoneContentCfg（企鹅空间动态配图）、KZoneAvatarCfg（头像）

> 注：PersonCfg（人物）和 EndingPartCfg（结局）走各自的专用编辑器，原版已自带图片预览，无需处理。

### 根因
- `group_right` 面板里只有 `txtex_desc`（悬浮说明文本），没有任何 Image/UISprite 组件。
- `Select()` 切换选中项时只刷新左侧属性列表，不涉及右侧图片。

### 修复（`ImagePreviewPatch`）
Postfix on `ModNormalEditView.Select`，支持 9 种配置类型：

| 配置类型 | 标题 | 取图字段 |
|---------|------|---------|
| CGCfg | CG 预览 | `urls[0]`（List） |
| BgCfg | 背景预览 | `url`（string） |
| ItemCfg | 物品预览 | `icon`（string） |
| BookCfg | 书籍预览 | `icon`（string） |
| RenshengguanMemoryCfg | 记忆预览 | `url[0]`（List） |
| ActionCfg | 行动预览 | `icon`（string） |
| PersonStateCfg | 状态预览 | `icon`（string） |
| KZoneContentCfg | 动态预览 | `imgs[0]`（List） |
| KZoneAvatarCfg | 头像预览 | `icon`（string） |

在 `group_right` 居中创建预览容器（标题 + 遮罩预览框 + 操作提示），选中条目时用 `UISprite.SetTextureUrl()` 加载图片（支持 "Mods/..." 路径）；带 `RectMask2D` 裁剪，附带"滚轮缩放 / 拖动平移 / 双击复位"交互。

### 布局修正
后续又针对预览位置做了两点修正：
- **标题贴住图片上方**：原先标题固定在容器顶部，图片等比缩放后如果没有占满高度，标题会与图片顶部产生空隙。现在由 `PreviewFit` 在图片加载/尺寸变化后计算缩放后图片高度，把标题底边贴到图片顶边。
- **底部上移半个字体高度**：预览框底部上移 `PreviewTitleFontSize * 0.5f`，避免图片抵到下方栏。

实现上抽出 `PreviewRootHeight / PreviewTitleHeight / PreviewTitleFontSize / PreviewBottomOffset / PreviewFrameHeight` 等常量；`PreviewFit.Update()` 同时监听 sprite 与 frame 尺寸变化，避免窗口布局变化后位置不重算。

### 关键实现细节
- 图片 RectTransform `pivot=(0,0)` 对齐游戏内部 sprite 的 `pivot=(0,0)`（`Sprite.Create` 统一左下角），避免 pivot 不匹配导致图片偏移/裁切。
- 居中靠 `Apply()` 计算 `anchoredPosition = -displaySize/2 + pan` 实现，不依赖 sprite pivot。
- 图片以原始分辨率 `sizeDelta` 设定，仅用 `localScale` 等比缩放，绝不拉伸/压扁宽高比。
- 默认缩放到刚好完整显示（`fit`），滚轮可放大至 `fit × 8`。

### 换图后预览不实时更新（热更新补丁，✅ 已修复）

**现象**：上传新图片或改完路径点「保存」后，预览仍是旧图，必须退出编辑器重进才能看到新图。

**根因**（两层）：
1. 预览原本只在**选中条目**（`Select`）时刷新；点「上传」只把新路径写进配置，点「保存」只把配置写进 JSON，两者都不触发 `Select`，预览没有刷新的机会。
2. 游戏对加载过的图片按路径做了内存缓存：`UISprite.SetExternTextureUrl` 遇到相同 URL 直接跳过，更深一层 `ResMgr.resDict` 里 `ResInfo.isLoaded` 会直接返回内存里的旧 Sprite。而上传是按**原文件名**覆盖复制（`Utils.CopyTo(..., overwrite: true)`），路径不变 → 缓存永远命中旧图。缓存只在界面全部关闭时（`UIMgr.CloseAllView` → `ResMgr.UnloadUnusedAssets`）清理，这正是"退出重进才生效"的原因。

**修复**：
- `ImagePreviewSaveRefreshPatch`：Postfix on `ModNormalEditView.OnClickSave`，保存后立即刷新预览。
- `ImagePreviewUploadRefreshPatch`：Prefix on `AtlasMgr.SelectImage` 包装回调，选完图片立即刷新预览（不用等保存）。仅在通用编辑器打开时生效（`UIMgr.GetOpeningView`），不影响封面上传等其他界面。
- 两处刷新均走**强制重读**：把 "Mods/..." 相对路径解析为磁盘绝对路径（`ModCtrl.GetFullUrl`），调用游戏自带的 `SetExternTextureUrl(full, _isReload: true)` 通道绕过上述两层缓存，从磁盘重新读图；重读后 `ResMgr` 缓存同步更新为新图。若磁盘上图片已被删除则清空预览。

## 5. 物品编辑器：持有上限不可见（✅ 已修复）

### 现象
MOD 作者反馈：编辑器没有设置物品持有量(maxcount)的字段，导致消耗品物品的修改有问题。

### 根因
`ItemCfg.maxcount` 带 `Hide=true`，通用编辑器(`ModNormalEditView`)会跳过该字段；`DefaultValue=1` 意味着 MOD 作者新建物品时该值固定为 1。
`BagMgr.AddItem` 用 `maxcount` 做 `Utils.Clamp` 上限，`UseItem` 每次减 1。对消耗品(type=1)影响最大——`maxcount=1` 时玩家最多持有 1 个，买第 2 个被截断，用掉后变 0。
原版消耗品的 maxcount 各不相同（食物类=2，奖券=10 等），作者无法在编辑器设置此值。

物品类型与 maxcount 的关系：

| type | 名称 | 原版 maxcount | 默认值 | 受影响 |
|------|------|--------------|--------|--------|
| 1 | 消耗品 | 2 / 10 等各不同 | 1 | **是，严重** |
| 2 | 珍视物品 | 全是 1 | 1 | 否 |
| 4 | 工具 | 全是 1 | 1 | 否 |

### 修复（`ItemMaxCountPatch`）
Postfix on `InitFields`（仅 ItemCfg）：
- 解除 `maxcount` 的隐藏（`attr.Hide=false`），并补充取值说明：
  > 物品的持有上限（背包里最多同时持有的数量）。
  > · 消耗品：建议按需设置（如食物类常设为 2~10），默认 1 表示只能持有 1 个
  > · 珍视物品 / 工具：通常为 1（装备型，不叠加）
  > 添加物品时若已有该物品，数量会被限制在此值以内。
- 调 `itemgroup_property.SetDatas` 刷新。

### 对玩家零依赖
产出的 mod 只是存了一个 `maxcount` 值，原版客户端 `BagMgr.AddItem` 直接读 `itemCfg.maxcount` 做 Clamp，不依赖本插件。

## 6. 企鹅空间编辑器增强（✅ 已增强）

### 现象
MOD 作者反馈企鹅空间相关表存在多类可用性问题：
- `KZoneContentCfg` 里**说说 / 日志没有显式区分**，作者只能靠 ID 规律猜；
- 日志的 `title` 字段没有 `[CfgProperty]`，编辑器里看不到「日志标题」；
- 正文和评论内容不按段落显示，`\n` 字面量会挤在一行里；
- 动态、评论、资料中大量字段只显示裸 ID，例如发布者、评论人物、点赞者、评论链、BGM 等，作者需要来回查表；
- 灰色说明过长时会超出右侧输入区域。

### 根因
- 游戏用 `KZoneContentCfg.id.ToString().StartsWith("99")` 隐式判断：
  - `99...`：日志（Blog）；
  - 其它：说说（Talk）。
  编辑器没有展示该规则。
- `KZoneContentCfg.title / visitCnt` 是公开字段但没有 `[CfgProperty]`，通用编辑器不会渲染。
- 通用编辑器 `Cell_ModNormalPropertyItemUI` 只提供单行式 `InputField`，不适合 QQ 空间长文本。
- 多个字段是跨表引用，但原版只显示原始数字。

### 修复 A：说说 / 日志可视化
补丁：`KZoneContentEditorFieldsPatch` + `KZoneContentListDisplayPatch`

- 左侧列表显示类型前缀：
  - `[301][说说]正文摘要`
  - `[9900301][日志]日志标题或正文摘要`
- 右侧 ID 字段动态显示为：
  - `ID（说说）`
  - `ID（日志）`
- 给 ID 字段悬浮说明补充规则：
  - ID 以 `99` 开头为日志；
  - 其它为说说。

### 修复 B：补出日志标题和浏览量
`KZoneContentCfg.title` 与 `visitCnt` 原本没有 `[CfgProperty]`，现在在 `InitFields` 后手动注入 `ModFieldItem`：

| 字段 | 编辑器显示 | 显示条件 / 说明 |
|------|------------|----------------|
| `title` | 日志标题 | 仅 ID 以 `99` 开头时显示；说说自动隐藏 |
| `visitCnt` | 浏览量 | `0`=游戏随机，`>0`=指定浏览量，`-1`=不显示浏览量 |

### 修复 C：正文 / 评论正文多行显示
补丁：`KZoneTextPropertyDisplayPatch`

- 对 `KZoneContentCfg.content` 和 `KZoneCommentCfg.content`：
  - `InputField.lineType = MultiLineNewline`；
  - 文本组件开启横向换行；
  - 把配置中的字面量 `\n / \r\n / \r` 转成真实换行显示。
- 输入框高度使用原 cell 布局数据扩展，避免长文本挤成一行。

### 修复 D：ID 字段灰字解析
补丁：`KZoneTextPropertyDisplayPatch` + `KZoneEditorTextUtil`

对以下字段在输入框下方显示灰色说明，直接把 ID 解析成人可读文本。

#### 企鹅动态 `KZoneContentCfg`

| 字段 | 原含义 | 灰字示例 |
|------|--------|----------|
| `role` | 发布者 ID | `对应人物：[蕾蕾（薛诗蕾）]` |
| `thumbs` | 自动点赞：`人物ID,延时;...` | `对应点赞：[201=梁超杰（立即）；202=冯馨月（3秒后）]` |
| `comments` | 自带评论 ID：`评论ID,延时;...` | `对应评论：[30101 麦彦华：好看吗？（立即）]` |
| `options` | 玩家评论 ID 列表 | `玩家评论：[30107 主角/玩家：想起它被老班没收的那天…]` |

#### 企鹅动态评论 `KZoneCommentCfg`

| 字段 | 原含义 | 灰字示例 |
|------|--------|----------|
| `roles` | 评论人物 ID；第 2 个存在时表示回复/提到对象 | `对应人物：[主角/玩家，蕾蕾（薛诗蕾）]` |
| `parent` | 根部评论 ID；`0`=直接回复动态 | `根部评论：[30101 麦彦华：好看吗？]` |
| `comments` | 自带回复 ID：`评论ID,延时;...` | `自带回复：[30104 主角/玩家→肖清雅：一点也不恐怖（3秒后）]` |
| `options` | 玩家回复 ID 列表 | `玩家回复：[30107 主角/玩家：……]` |

#### 企鹅空间资料 `KZoneProfileCfg`

| 字段 | 原含义 | 灰字示例 |
|------|--------|----------|
| `id` | 资料对应的人物 ID | `资料人物：[蕾蕾（薛诗蕾）]` |
| `bgm` | 背景音乐，对应 `AudioCfg.id` | `对应音乐：[42001=音乐名]`，`0` 显示为 `无` |

人物名解析规则：
- `0` → `主角/玩家`；
- `-1` → `当前空间主人/对方`；
- 其它 ID 优先使用 `KZoneProfileCfg.name`，若与 `PersonCfg.name` 不同，则显示为 `空间名（人物本名）`，例如 `蕾蕾（薛诗蕾）`；
- 找不到时显示 `未知ID:xxx`。

评论摘要解析规则：
- 读取 `Cfg.KZoneCommentCfgMap[id]`；
- 用 `roles[0]` 显示评论者；若 `roles[1] >= 0`，显示为 `评论者→被回复者`；
- 拼接评论正文摘要；正文过长时截断。

### 修复 E：灰字换行与间距
灰字说明长时会超出右侧输入区域，因此增加了手动换行和布局保护：

- **每 27 个可见字符插入一次换行**，前缀也计入，例如 `玩家回复：` 同样参与计数；
- 避免把 `<sprite=...>` 富文本标签从中间切断；这类标签按 1 个可见字符计数；
- 多行灰字使用 `TextAnchor.UpperLeft`，并按输入框下沿重新定位；
- 多行时同步调整提示文本框高度和透明 spacer 高度，避免灰字顶到输入框或被下一字段遮住。

### 间距实现方法
一开始尝试过直接增高 `Cell_ModNormalPropertyItemUI` 的 `RectTransform / LayoutElement`，但这会让「装 ID 的输入框」本体看起来变高，不符合预期。

最终实现改为：
1. 灰字 `Text` 仍作为当前 cell 的子对象，固定显示在输入框下方；
2. 在当前 cell 后面插入一个独立、透明的 `KZoneRoleGapSpacer`；
3. spacer 带 `RectTransform + LayoutElement`，只负责把下一项往下推开；
4. cell 复用 / 隐藏 / 销毁时由 `KZoneRoleGapCleanup` 自动销毁 spacer，避免残留；
5. 单行和多行都会记录基础 spacer 高度，多行时按行数扩大。

这样视觉效果是：
```text
字段输入框
灰字说明第 1 行
灰字说明第 2 行
<透明 spacer>
下一个字段
```
而不是把输入框本身拉高。

## 7. 送礼触发事件：编辑器完全无法编辑（✅ 已修复）

### 现象
MOD 作者反馈：编辑器完全无法做送礼触发事件。

### 根因
送礼触发事件使用独立的 `GiftEvtCfg` 表（与 `EvtCfg` 分开），通过 `BagMgr.GiveItem → TryShowGiftEvt → GetGiftEvtCfgId` 匹配 `物品 + NPC + 条件` 触发对话。
该表有**两层缺陷**，导致编辑器完全不可编辑：

**缺陷1：没有 `[CfgClass]` → 不出现在配置类型列表**
编辑器 `ModPageUploadView.OnOpen` 扫描带 `[CfgClass]` 且 `Order!=0` 的类型构建列表，`GiftEvtCfg` 没有此特性 → 作者看不到入口。

**缺陷2：字段没有 `[CfgProperty]` → 打开后全空白**
`ModNormalEditView.InitFields` 只收集带 `[CfgProperty]` 的字段，`GiftEvtCfg` 7 个字段全部是裸字段 → 编辑器空白。

### 修复（`NativeCfgPatch.cs` 通用框架，GiftEvtCfg 作为注册类型之一）

**Patch A：让送礼事件出现在配置列表**
- `GiftEvtCfgTypePatch`（Postfix on `ModPageUploadView.OnOpen`）：将 `GiftEvtCfg` 插入 `allTypes`（按 Order 排到 BookCfg 和 ActionCfg 之间）。
- `GiftEvtCfgRenderPatch`（Prefix on `ModPageUploadView.OnRenderItem`）：原版用 `GetCustomAttribute<CfgClassAttribute>().Name` 取显示名，`GiftEvtCfg` 没有特性会 NullRef。Prefix 拦截其渲染，直接设置名称为「送礼触发事件」并跳过原方法。

**Patch B：注入可编辑字段**
`GiftEvtCfgFieldsPatch`（Postfix on `ModNormalEditView.InitFields`，仅 GiftEvtCfg），手动为 7 个字段构造 `ModFieldItem`：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 事件ID（必填） |
| item | int | 物品ID（必填，下拉选物品） |
| npc | List\<int\> | 可送礼NPC列表，逗号分隔 |
| cond | List\<List\<double\>\> | 前提条件 |
| talkId | List\<List\<int\>\> | 对话ID，按NPC顺序对应 |
| type | List\<int\> | 0=转移物品，1=不转移 |
| redpoint | int | 1=红点提示，0=无（默认1） |

`item` 字段的 Range 设为 `CfgPropertyRange(typeof(ItemCfg))`（不限 id 范围），因为 GiftEvtCfg 的 item 可引用 ItemCfg（物品）或 BookCfg（书籍），id 跨 1xxx-8xxx 多个段，设窄了反而导致已有数据在下拉框中选不到。

**Patch C：左栏条目名称**
`GiftEvtCfgItemNamePatch`（Postfix on `ModNormalEditView.OnRenderItem`，仅 GiftEvtCfg）。原版从 `nameFields={name,title,content,text}` 取名称，GiftEvtCfg 没有其中任何字段 → 全部显示「新建」。Patch 用 `item + npc` 拼出有意义的名称，如 `[1]一盒草莓→NPC102`。物品名优先从 ItemCfgMap / BookCfgMap 取。

**Patch D：物品下拉框追加书籍**
`GiftEvtCfgItemDropdownPatch`（Postfix on `ModNormalEditView.OnRenderProperty`，仅 GiftEvtCfg 的 item 字段）。原版下拉框只从 `ItemCfgMap` 拉取，书籍在 `BookCfgMap` 中选不到。Patch 在原方法构建好选项列表后追加 BookCfg 条目，以「（书）」后缀标注，如 `[3012]时间简史（书）`。

### 字段说明文案
各字段的悬浮说明面向 MOD 作者，使用简洁规范的书面语，避免开发者术语和过于口语化的表述。关键格式说明：
- **NPC ID**：提示下方对话ID和类型标记均按此列表顺序一一对应。
- **对话ID**：说明外层分号分隔不同NPC、内层逗号分隔同一NPC多段对话，并给出三种例子。
- **类型标记**：说明0=消失（赠予NPC）、1=不消失、留空等同0，多个NPC时按顺序对应。
- **物品ID**：提示下拉列表同时包含物品和书籍，书籍以「（书）」后缀标注。

- **Patch A2**（`OnRenderItem` Prefix）：拦截注册表类型的渲染，避免 `CfgClassAttribute.Name` NullRef。
- **Patch B**（`InitFields` Postfix）：按注册的字段定义批量构造 `ModFieldItem`。
- **Patch C**（`OnRenderItem` Postfix）：用 `[id]name` 或自定义回调拼左栏名称。
- **Patch D**（`OnRenderProperty` Postfix）：可选回调，GiftEvtCfg 用它实现 item 下拉框追加书籍。

原 `GiftEvtPatch.cs`（355 行）已删除，功能全部迁移到框架。送礼事件的特殊逻辑（左栏名称拼 `[id]物品名→NPC`、item 下拉框追加书籍）通过 `FormatItemName` 和 `OnRenderPropertyHook` 两个可选回调保留，行为与之前完全一致。

### 对玩家零依赖

**B. 通用编辑器搜索**
- Postfix on `OnOpen`：在列表滚动区上方创建搜索栏，输入文字后按 ID 或名称（name/title/content/text 字段）过滤。

- Postfix on `OnClickNewItem` / `DeleteCur`：新增或删除条目时自动清除搜索，让新条目立即可见。

## 10. 对话/选项编辑器：无法插入小游戏及胜负分支（✅ 已修复）

### 现象
MOD 作者反馈：在对话编辑中无法插入小游戏（辩论、砍价、打架、数独等），也无法根据其胜负判定后续触发的对话。

### 根因（已在代码 + 运行时双重验证）
游戏运行时**完整支持**在对话/选项/事件中插入小游戏，并根据胜负跳转后续对话：
- `TalkCfg.miniGame` / `OptionCfg.miniGame` 都是 `List<double>` 字段，`miniGame[0]` 是小游戏编号，`miniGame[1..]` 是传给小游戏的参数。
- 运行时 `NewTalkView.NextTalk()` → `CommonEvtMgr.JumpToMiniGame()` → `FuncMgr.OpenMiniGame()`，按编号打开对应小游戏界面（6=辩论、5=数独、21=打架等，共 20+ 种）。
- 小游戏结束后，在 `CloseView` 里按胜负分支：
  - **胜利** → 执行 `effect` → 跳转 `nextTalk`（或选项的 `talkId`）
  - **失败** → 执行 `effect2` → 跳转 `nextTalk2`（或选项的 `talkId2`）

但两个编辑器都**没有 `miniGame` 字段的编辑入口**：

| 编辑器 | 配置类 | 原因 |
|--------|--------|------|
| 对话编辑器 `ModEvtEditView` | `TalkCfg` | 硬编码 UI，prefab 里没有 miniGame 输入框 |
| 选项编辑器 `ModEvtOptionView` | `OptionCfg` | 同上 |

而胜负分支的 UI（条件判断开关后的两组后续对话输入框）**原本就存在**，只是标签没提到"胜负"，作者不知道它们同时承担胜负分支的语义。

### 运行时的五个硬坑（决定了编辑器必须做校验）

初版补丁只加了输入框，实测后发现"能填"不等于"能用"。逐行审查运行时代码确认了四个坑，作者若不知情，产出的 mod 会在玩家侧出问题：

1. **无效编号会卡死游戏**。`FuncMgr.OpenMiniGame` 的 switch 没有 12/25/38/40 的分支（这些编号在配置表里存在但没有实现），编号超出 1~48 还会在结尾查 `Cfg.MinigameCfgMap` 时抛异常。而进小游戏前对话界面已被 `JumpToMiniGame` 隐藏——结果就是黑屏卡死。
2. **编号 1/2/4/28 不接回对话**。考试、抢红包、直接结算这四种走的是无参重载，结束后不会执行胜负分支，对话链就断了。
3. **失败分支缺失会空引用卡死**。小游戏 `CloseView` 的失败分支最后是裸的 `fail()`（没有判空），而从对话/选项进入时 `fail` 恒为 null：
   - 选项：`talkId2` 为空 → `GetNextTalk2()` 返回 0（**不回退**）→ `fail()` 空引用 → 卡死；
   - 对话：`nextTalk2` 为空会回退到 `nextTalk`（胜负同路，作者以为分支没生效）；若 `nextTalk` 也为空 → 同样 `fail()` 卡死。
4. **效果会重复执行**。设了小游戏的对话/选项，其 `effect` 在胜利后会（再）执行一次、`effect2` 在失败后执行；选项还会在点击时先按 check 跑一遍 effect。属性变化写在本条上容易加两次。
5. **每个小游戏对"对话/选项"的支持面完全不同**（第二轮全量审查新发现，见下节）。

### 第二轮全量审查：40+ 个小游戏界面逐个核对的支持矩阵

每个小游戏界面的 `OnOpen` / `CloseView` 都是独立实现。逐个审查后发现三类新的"无法运行"级问题：

**A. 只支持选项、不支持对话的编号**——`CloseView` 只写了 `MiniGameFromType.Option` 分支，从对话（Talk）触发时结束后什么都不做，而对话界面还处于隐藏状态 → 黑屏卡死：

> 17 知识竞赛、18 辩论赛、19 长跑、27 跑步、29 大头贴、34 狂点蓄力、36 辩论选队友、37 爱心早餐、42 漫展、44 生日派对、47 世博会

（其中 29 大头贴更特殊：选项分支也只跳 `talkId` 不分胜负。）

**B. 对话和选项都不能用的编号**：
- 3 占卜：结束走 `EndGame` 社交小游戏流程，不接对话链；
- 33 学习卡牌：`OnOpen` 无条件执行 `levels[typeId-1]`（levels 只有 3 项，而 typeId 是对话/选项 ID，动辄七八位数）→ 必崩。它的 `CloseView` 虽有 Talk 分支，但实际是永远走不到的死代码；
- 30 情侣画、31 祈福牌、39 对话框：恋爱/特殊流程专用。

**C. 只填编号（不带参数）就会在打开时报错卡死的编号**——`OnOpen` 对参数列表不判空直接取值（`list.Count` / `list[0]` / 查表）：

| 编号 | 崩溃点 | 必需参数 |
|------|--------|---------|
| 10 打砖块 / 20 戳指缝 / 24 弹钢琴 / 46 编织 / 48 简笔画 | `list.Count` 空引用 | 至少 1 个（难度/配置编号） |
| 11 篮球 / 26·32 羽毛球 | `list[0]`、`list[1]` | 先手(0/1) + 比赛/等级编号 |

## 11. 事件对话编辑器：人物预览槽位 3 → 9（✅ 已修复）

### 功能说明
在事件对话编辑器中，画面下方的人物预览区分为左、中、右三个方位，每个方位有 3 个人物位，全屏最多同时摆放 9 个人物：
- 点击空位上的 **+**：为当前对话添加一个入场人物（人物会出现在你点击的那个位置）；
- 点击已有人物的位置：可以把这个人换成别人，或选「清除」让他退场；
- 添加/移除的结果会同步写入下方的「人物动作」文本框（roles 字段），也可以继续手动编辑该文本框做进阶调整。

### 原版的问题
游戏预制体里其实一直有 9 个人物位（每个方位 3 个），但原版代码只启用了每个方位的第 1 个位子：
- 同一方位添加第 2 个人物时会被静默忽略，界面上最多只能看到 3 个人物；
- 剩下 6 个位子从未启用，而且是"半成品"：+ 按钮的位置、点击范围都没有调好。

### 初版补丁遗留的三个问题（本次修复）
初版只是把 6 个隐藏位子创建了出来，实测发现三个问题，根源都在预制体的"半成品"属性上：

| 现象 | 根源 |
|------|------|
| 明明有很多空位，"+" 却只显示 3 个；加完一个人，旁边才冒出下一个 "+" | 初版沿用了原版的"链式显示"：前一个位子有人，下一个位子才出现。已改为 9 个位子全部常显 |
| 点了 "+" 没反应，或者人物加到了别的位置 | 两个原因：① 9 个位子排得很密（相邻中心相距最小仅 140 像素），而每个 + 按钮的隐形点击范围宽达 400 像素，相邻位子的点击范围大面积互相叠压——点这里的 +，响应的其实是隔壁的按钮，人物因此加错方位；② 人物落位由代码按内部编号顺序分配，与你点击的位置无关。现已把点击范围收窄到 + 图标附近（互不重叠），并保证人物就落在你点击的那个位子 |
| 清除一个人物后，其他人跳到别的位置（错位） | 预制体位子的内部编号顺序（_0 → _1 → _2）与屏幕上的实际排列（_2 最左、_0 居中、_1 在右）不一致，代码按编号顺序填充导致人物在屏幕上乱跳。现已按屏幕位置从左到右填充，并且增删人物时，已在场的人保持原位不动 |

另外还修正了两个细节：
- 6 个新位子的 + 图标高度与原有位子不一致（一个居中、一个偏下），已统一对齐；
- 替换一个"从上文对话入场"的人物时，原逻辑只加新人不送走旧人，会变成两个人同时在场。现在会自动给旧人补一条退场动作，实现真正的"换人"。

### 补充修复：清空后重新添加同一人物没反应
实测发现：把某个人物清空后，再点 +（无论哪个位置）把他加回来，人物不会出现。

原因：清空会在当前对话里留下一条"退场"记录（`[人物ID, 2002]`，与原版一致），重新添加时又在后面追加一条"入场"记录（`[人物ID, 1002, …]`）。而编辑器判断"谁在场"时，对同一个人只认列表里**靠前的那条动作**——退场在前、入场在后，入场就被忽略了，这个人被当成"已退场"。（原版编辑器同样存在此问题，只是原版每方位只有一个位子，很少有人触发。）

修复：每次通过 + 添加或替换人物时，先清掉此人在当前对话里的残留动作记录（主要就是那条退场记录），再写入新的入场记录。这样"清空 → 重新添加"、"清空多人 → 逐个加回"都能正常工作，写入的数据也更干净（一个人不会同时挂着退场和入场两条记录）。

### 实现（`EvtRoleDisplayPatch`）
- **补丁 A**（`InitUI` Postfix）：创建 `root_left/mid/right_1、_2` 共 6 个槽位 Cell；把它们的 + 按钮布局对齐到 `_0` 槽样式；将全部 9 个槽位的点击判定区收窄到 + 图标附近（130×130，人物立绘不参与点击判定，不受影响）；每个方位的槽位列表按屏幕 x 坐标排序。
- **补丁 B**（`RefreshRoles` Prefix）：替换人物分配逻辑，优先级为：刚点击添加的人物 → 落在被点击的槽位；已在场人物 → 保持原槽位；其余 → 按"居中 → 左 → 右"填空位（每方位第 1 人显示在原版编辑器的居中位置）。最后 9 个槽位全部显示。
- **补丁 C**（`OnCreateRole` Prefix）：替换 + 按钮逻辑，允许同方位多人；记录被点击的槽位供补丁 B 精确落位；替换上文入场的人物时自动补退场动作。

> 预制体关键数据（UnityPy 实测）：9 个槽位 x 坐标依次为 -770/-560/-350（左）、-210/0/210（中）、350/560/770（右)，即每方位内视觉顺序为 _2、_0、_1；_1/_2 槽默认 inactive，btn_add 为半成品布局。

### 向后兼容（旧 mod / 旧数据）
补丁不会自动改写任何 mod 数据：打开、浏览、预览对话都只是读取；只有作者主动点 +、换人、清除时才写入 `roles` 字段，写入格式与原版编辑器完全相同。玩家游玩时的人物站位由游戏运行时决定，与编辑器无关。

用工坊大型 mod（含 6040 条对话）实测验证过显示兼容性：
- 每方位只有 1 人的对话（最常见）：显示在居中位置，与原版编辑器完全一致；
- 同方位 2~3 人（原版编辑器只能显示第 1 个）：现在全部可见，属于纯增强；
- 同方位 4 人以上（手写 JSON 才能做出的数据）：编辑器最多预览 3 个，放不下的不显示，但数据和游戏运行完全不受影响。

### 对玩家零依赖
补丁只影响编辑器。人物入场/退场数据写入对话的 `roles` 字段（标准 JSON），原版客户端的对话界面本身就支持同方位多人（`NewTalkView.posRoles` 按方位维护人物列表），玩家无需安装本插件。

## 12. 动作指令可视化：悬浮翻译 + 移动位置示意 + 预览本句（✅ 已实现，待实测）

### 现象
MOD 作者反馈：编辑事件对话时缺少可视化——动作指令是一串数字（如 `1,3004,300`），既看不懂每个数字是什么意思，也看不出人物移动 300 到底是多远，只能反复打开剧情预览来确认。

### 根因
- "动作指令"输入框里的内容是 `roles` 字段的原始数字串，游戏内部有 30 多种动作编码（进场、退场、横移、跳跃、换装、表情……），编辑器不做任何翻译，也不提示每种动作各有哪些参数；"屏幕效果"输入框同样是裸数字。
- 编辑器右侧的人物预览只认三类信息：谁进场（站左/中/右）、谁退场、穿哪套服装。其余动作一概不体现——尤其是横移/纵移，人物在预览里纹丝不动。
- 原版的剧情预览入口在外层事件编辑页，且**永远从事件第一句开始播**（`PreviewTalkView.OnOpen` 写死取 `talkId[0]`）。调第 40 句的动画就得从第 1 句一路点过去，改一次看一次。

### 改进（`EvtActionVisualPatch`，纯显示层，不改动任何数据格式）

**1. 动作指令悬浮翻译**
鼠标移到动作指令输入框上时，输入框下方弹出提示框，把数字串逐条翻译成人话，例如：

> 小明：横向移动 (向右300、用时0.5秒)
> 小红：淡出退场

- 人名从人物配置自动解析，取不到时显示"人物+编号"；
- 高频动作（进场/退场/横移/纵移/跳跃/抖动/放大/换装/表情等）会把参数也翻译成"延迟x秒""站左侧"这类说明；
- 不认识的编码显示"未知动作(编码)"，方便发现笔误；
- 当前对话没填时提示"（当前对话没有动作指令）"；格式打错时提示"内容格式有误"，不影响编辑器其他功能；
- 编辑完按回车时若鼠标还悬在输入框上，提示内容立即同步更新。

> 迭代记录（两轮用户实测反馈）：
> ① 初版做成输入框下方的常驻灰字 → 被紧邻的"屏幕效果"输入框遮住看不清（group_content 内控件排布很密）→ 改为悬浮提示；
> ② 第二版向上弹出 → 内容多时溢出屏幕上边缘 → 改为**向下弹出 + 屏幕边缘保护**（底部放不下自动上抬、右侧超界自动左移）。提示框置顶渲染，盖住下方控件无妨——鼠标正悬在输入框上，看的就是提示本身。

**2. 屏幕效果悬浮翻译**
"屏幕效果"输入框加了同样的悬浮翻译：显示CG/漫画分格/屏幕震动/手机界面等 20 余种效果码连同参数一起翻译（如"显示CG (CG编号12)"）。

**3. 移动位置示意**
当前选中的对话里如果给人物填了横向/纵向移动指令，右侧人物预览会**直接把这个人物挪到移动后的位置**——填 300 挪多远、改成 500 挪多远，一眼可见。

- 预览画面是真实对话画面的 0.7 倍缩放，位移按同比例换算，与真实效果的比例一致；
- 只反映**当前选中的这一句对话**的移动。切到别的对话时，人物回到标准站位（之前对话的移动不会累计——预览槽位与游戏的连续坐标本就不完全相同，累计只会造成误导）；
- 移动的过程动画（速度、延迟）仍需预览查看，本功能展示的是移动终点。

**4. "预览本句"按钮**
对话编辑器右侧按钮列新增"预览本句"（选中对话后出现）：点击后直接从**当前选中的这句对话**开始播放剧情预览，动作动画按真实游戏方式演出，右键关闭返回编辑器。

- 传入的是编辑器内存中的最新数据——**没保存的修改也能立即预览**；
- 从本句开始播放后，点击画面会**继续沿剧情往后走**（方便看本句与后续的衔接，经用户确认保持此行为），右键随时关闭；
- **自动补齐起始状态**（用户实测反馈"时而没背景、人物占位乱"后新增）：本句没填背景时沿剧情链回溯继承背景（mod 通常只在第一句/换场景句填背景）；更早对话里进场的人物自动"放置"到对应方位，本句动作不再作用在不在场的人身上。补齐只发生在传给预览器的副本上，**编辑器数据零改动**；
- **前文状态完整还原**（第二轮增强，用户确认需要）：在补站位的基础上，沿剧情链倒序回溯每个人物的**服装、发型、姿势、剪影、朝向（翻转次数奇偶）、放大、累计位移**，合成对应指令注入起始句副本——本句预览时人物穿着、站位与真实游玩一致。回溯遇到该人物的进场/退场即停（更早的状态属于上一个出场周期）；回溯失败自动回退为"仅补站位"，预览永不因此打不开；
- 三层递进的调试节奏：舞台位移示意（零成本看终点）→ 预览本句（一键看本句动画）→ 原版事件预览（从头看整体流程）；
- 如实残留的三个小限制：①纵向累计位移以 0.4 秒快速滑动呈现（引擎的纵移指令没有时长参数，横移可 0.011 秒瞬移）；②表情/跳跃/抖动等瞬时动画不回放（真实游戏里本就不跨句残留）；③前文有累计位移**且**本句又有移动指令时，本句移动动画的起点会从标准站位而非前文位置开始（引擎对同句多条移动的补间竞争所致），**移动终点始终正确**。另外若到达本句的路径有多条分支，回溯沿"第一个反查到的前驱"单链还原（与编辑器小舞台同规则）。首次使用时会 Toast 提示一次。
- 代码审查补充的两项修正：①传给预览器的**每一句**都改为 roles 列表浅拷贝的副本——预览播放器每句播放前会对 roles 列表就地排序（进场条目提前），直接传编辑器内存对象会悄悄重排作者在动作指令框里看到的数字串顺序；②状态回溯按"进场条目最先执行"的引擎排序语义分两遍处理，作者把换装等指令写在进场指令前面（数据顺序）时不再漏算。

### 实现要点
- **补丁 1**（`InitUI` Postfix，悬浮翻译）：用游戏自带的 `AddMouseEnter/AddMouseExit` 扩展（`Sdk.GameObjectExtension`，原版未占用这两个输入框的回调）注册悬浮/移出；提示框为深色半透明背景 + 白字，尺寸按 `Text.preferredWidth/Height` 自适应（不用 ContentSizeFitter——动态创建的对象上不稳定）；定位先取输入框左下角世界坐标，再经 `ScreenPointToLocalPointInRectangle` 转到编辑器根节点本地坐标系做边缘钳制，不受分辨率与画布缩放影响；整体 `raycastTarget=false` 避免遮住输入框触发"移出"闪烁。另给动作指令输入框追加 `onEndEdit` 监听——排在原版解析逻辑之后，触发 `RefreshRoles()` 让位移示意即时生效（原版编辑动作指令后不刷新舞台）。
- **补丁 2**（`OnRenderRole` Postfix，位移示意）：渲染立绘后，累加当前对话里该人物所有横移(3004)/纵移(3008)的偏移量 × 0.7，加到立绘的基准位置上；无移动指令时归零还原（立绘对象会被复用，必须每次重置）。
- **补丁 3**（`InitUI` Postfix，预览本句）：克隆"保存"按钮生成新按钮——`group_btn` 带 `VerticalLayoutGroup`（UnityPy dump 核实），克隆体会被自动排版，无需手算坐标；点击后把编辑器内存数据组装成 `PreviewTalkView` 的参数（结构与原版 `ModPreviewTipsView.OnClickOK` 一致），起始句列表只放当前对话 ID。数据来源细节：`personCfgs`/`audioCfgs` 编辑器加载时已合并原版直接传（`audioCfgs` 懒加载需先调 `LoadAudioCfg`）；`customBgCfgs`/`customCGCfgs` 仅含 mod 自定义项，需与原版配置表合并后再传（`PreviewTalkView` 收到非空表后不再回退原版表）。
- **补丁 4**（`Select` Postfix）：选中状态变化时隐藏悬浮提示（防旧内容残留）+ 联动"预览本句"按钮显隐（与原版删除按钮的显隐逻辑一致）。
- 动作中文名在插件内硬编码（游戏的 `TalkAnimeCfg` 配置表只有编号和类型，没有名称字段）；参数含义逐条对照游戏对话播放逻辑（`NewTalkView.HelpCheckRoleAction` 及 `screenEffect` 各分支）核实。
- 0.7 缩放比依据：真实画面同方位多人间距 300px ↔ 编辑器槽位间距 210px，立绘缩放也是 0.7。
- 与修复 11（`EvtRoleDisplayPatch`）兼容：其 `RefreshRoles` Prefix 替换原方法后，Harmony 仍会执行本补丁的 Postfix，两者互不干扰。

### 对玩家零依赖
四个功能都只改编辑器的显示与预览入口，不写入任何数据。产出的 mod 与原版格式完全一致，玩家无需安装本插件。
